# CV — carte de mise en page (haut → bas, par régime)

> **État du snapshot** : généré par audit automatisé du code (12 agents). Trois PR ouvertes modifient certains blocs — **#154** (tailles en-tête → bloc 1), **#155** (entrées « entreprise / poste / ville » sur 1 ligne → bloc 6), **#156** (dates Learnings & Loisirs → blocs 9-10) — s’y référer pour l’état cible de ces blocs.

Ce document est la carte **structurelle et ordonnée** du rendu du CV : il parcourt les blocs **de haut en bas** (tokens globaux → en-tête → … → loisirs) et donne, pour chaque bloc, les valeurs exactes dans les 4 régimes de rendu, avec citations `fichier:ligne`. Il **complète** [`docs/cv-rendering-rules.md`](cv-rendering-rules.md), qui est thématique (une règle par sujet) : ici l'entrée est l'**ordre de la page**, pas le thème. À utiliser pour la relecture/analyse (humaine ou LLM) d'un rendu de bout en bout et pour localiser une divergence à sa position dans le flux. La liste consolidée de toutes les divergences web ≠ PDF connues est en fin de document.

## Légende — les 4 régimes

| Régime                         | Déclencheur                                                      | Où s'applique la couche CSS                                                         |
| ------------------------------ | ---------------------------------------------------------------- | ----------------------------------------------------------------------------------- |
| **web mobile (<768)**          | vue normale, viewport `< 768px`                                  | classes de base Tailwind (sans `md:`)                                               |
| **web desktop (≥768)**         | vue normale, viewport `≥ 768px`                                  | classes de base + variantes `md:`                                                   |
| **PDF print**                  | impression réelle (`Cmd+P` / `window.print()` via `CvAutoprint`) | `@media print` + utilitaires `print:`                                               |
| **aperçu (.cv-print-preview)** | `?print=1` (et pendant `Cmd+P`)                                  | classe `.cv-print-preview` sur `<html>` — **n'active PAS** les utilitaires `print:` |

Rappels transverses :

- **WYSIWYG visé** : `web == aperçu ?print == PDF`. Piège récurrent : une règle « print » doit souvent être écrite DEUX fois (`@media print` **ET** `.cv-print-preview`) ; les utilitaires `print:` de Tailwind **ne s'appliquent pas** sous `.cv-print-preview` → l'aperçu peut mentir sur le PDF.
- **2 layouts distincts** : CV complet (`OfferTailoredShell`) et CV court (`CompactCvLayout`, route `/[lang]/short`). Le PDF s'imprime à ~786px (`@page { margin:10mm 1mm }` ≈ 208mm, `styles/globals.css:654-657`) → `@media (min-width:768px)` (`md:`) est **actif en même temps** que `@media print`, et dans le CSS compilé `md:` cascade **après** `print:` → à spécificité égale, `md:` gagne et neutralise le `print:` jumeau.
- **Le CV court n'affiche PAS** le job-fit détaillé (`variant="full"`), les entrées Études/Projets « longues » (`.cv-entry`), ni la section Loisirs : c'est un layout compact et tronqué (`SHORT_PROFILE_MATCH_MAX`, `Project`/`Study` compact). Sur **mobile (<768), `/short` rend le CV COMPLET** (`app/[lang]/short/page.tsx:236-242`) — l'instance courte n'apparaît jamais sous 768px.
- **`.cv-print-preview` est PERMANENT sur `/short`** (`short/page.tsx:193,240`) → sur ce layout « web desktop » == régime aperçu.

## 0. Tokens & régimes globaux

Deux tokens d'espacement pilotent tout le rythme du CV, définis dans `styles/globals.css` et consommés via `gap-[var(--cv-section-gap)]` (`components/CompactCvLayout.tsx:164`) et `.cv-section-body-gap { margin-top: var(--cv-section-body-gap) }` (`styles/globals.css:213-215`). Piège central : `@media print` et `.cv-print-preview` posent les valeurs sur `:root`, mais `.cv-full-cv-print-root` et `.cv-short-page` **re-déclarent** les tokens sur l'élément wrapper → cette valeur **masque** la couche `:root` (donc les régimes print/aperçu) pour tous leurs descendants.

**Règles / pièges**

- Sémantique des tokens : `--cv-section-gap` = espace INTER-section (au-dessus du titre, porté par flex-gap ou `margin-top`), `--cv-section-body-gap` = espace titre `h2 border-b` → contenu. Valeur UNIQUE par portée, jamais de marge par section en dur (évite le margin-collapse). Cf. `styles/globals.css:14-33`.
- Cascade décisive : parce que `.cv-full-cv-print-root` (`globals.css:92-95`) et `.cv-short-page` (`globals.css:52-61`) re-déclarent les tokens sur le wrapper lui-même, leur valeur l'emporte sur `:root` pour tous les descendants → **corps complet = 1.8/0.5 dans les 4 régimes**, **corps court = 0.9/0.4 dans les 4 régimes**. Les valeurs print/aperçu de `:root` (1.3/0.5) n'atteignent QUE ce qui est hors des deux wrappers.
- Flux « header hors `.cv-full-cv-print-root` » : dans `OfferTailoredShell`, `<Headers>` est rendu AVANT `<div className="cv-full-cv-print-root">` (`components/OfferTailoredShell.tsx:112-122`) → l'en-tête hérite de la couche `:root` et reste donc le seul élément du complet soumis à la divergence web(1.5/1) ↔ print/aperçu(1.3/0.5). Les sections du corps (About, Domains, JobFit, Contact, Jobs) sont dans le wrapper → uniformes.
- Invariant #2 respecté (pas de règle print orpheline) : `@media print { :root … }` (`globals.css:38-43`) et `.cv-print-preview { … }` (`globals.css:44-47`) sont des copies STRICTEMENT identiques → au niveau tokens, **aperçu == PDF**. Ici la parité passe par la classe jumelle, pas par des variantes `print:` (qui ne s'appliquent pas en aperçu).
- `.cv-short-page` porte aussi `zoom: var(--cv-zoom,1)` (écran, curseur `CvZoomSlider`) neutralisé `zoom:1 !important` en `@media print` (`globals.css:59-66`) ; le court n'est jamais rendu <768px (`app/[lang]/short/page.tsx:236-242`), donc le zoom n'existe que desktop/aperçu.

Sous 768px il est neutralisé → le court retombe sur la typo mobile lisible (invariant #3).

## ⚠️ Divergences web ≠ PDF connues

Classées par bloc, dans l'ordre de la page. **[V]** = divergence voulue/device-spécifique (WYSIWYG print-ish tenu), **[≠]** = vraie dérive aperçu≠PDF ou intention≠rendu, **[†]** = code mort/legacy.

### 0. Tokens & régimes globaux

- **[V]** `:root` diverge web (`--cv-section-gap:1.5rem` / `--cv-section-body-gap:1rem`, `globals.css:20,28`) vs PDF & aperçu (1.3rem / 0.5rem, `globals.css:40-41,45-46`) ; divergence RÉELLE mais confinée aux éléments hors wrappers (essentiellement l'en-tête du complet, `OfferTailoredShell.tsx:112-122`), car les corps complet/court re-déclarent les tokens et masquent cette couche.
- **[V]** `/short` device-spécifique : en mobile (<768) la page rend le CV complet → rythme `.cv-full-cv-print-root` = 1.8rem/0.5rem, alors qu'en desktop/impression elle rend le CV court `.cv-short-page` = 0.9rem/0.4rem (`short/page.tsx:236-242` ; `globals.css:57-58,93-94`) — deux layouts distincts, pas une régression.

### 1. En-tête

- **[≠]** CV complet — rôle : aperçu/web-desktop = `md:text-3xl` 30px (défaut) ou `md:text-2xl` 24px (`?photo`) mais PDF = `print:text-2xl` 24px / `print:text-xl` 20px ; les `print:` n'agissent pas en aperçu et aucun filet `.cv-print-preview` ne compense (`HeaderContent.tsx:154-161`).
- **[≠]** CV complet — âge : aperçu/web-desktop = `md:text-xl` 20px vs PDF = `print:text-base` 16px (`HeaderContent.tsx:173`).
- **[≠]** CV complet — padding-top de `.header-content` : web-desktop & aperçu = `md:py-12` 48px, PDF = `print:!pt-2` 8px (clearance toolbar retirée en PDF) → aperçu≠PDF sur l'espace au-dessus (`HeaderContent.tsx:68`).
- **[≠]** CV complet — nom au-delà de lg (fenêtre ≥1024) en aperçu : `lg:text-6xl/7xl` 60/72px vs PDF `text-4xl/5xl` 36/48px, idem avatar `lg:h-36` 144px vs `print:h-28` 112px et gap `md:gap-6` 24px vs `print:gap-4` 16px (`HeaderContent.tsx:142-143, :109, :71`).
